# N6: QA 评估

AI 动态决策是否触发 `tc-qa-engineer`，不按固定间隔。

## 进入消息

```text
🧪 N6 QA 评估 — Task {T-编号}
```

## 评分（1-5 分，总分 ≥ 8 触发）

| 维度 | 1 分 | 5 分 |
| ---- | ---- | ---- |
| 变更范围 | 单文件小改动 | 跨多模块/多项目 |
| 风险等级 | 纯 UI 文案 | 数据库/支付/认证 |
| 累积变更 | 上次 QA 后 1 个 task | 上次 QA 后 5+ 个 task |
| 功能边界 | 模块内部实现 | 完整用户可感知功能 |

## 必须触发（无需打分）

- 当前 feature 所有 task 完成
- API 接口变更
- 数据库 migration
- 认证/授权/支付逻辑
- 连续 5 个 task 未触发过 QA

## 跳过条件

- 纯文档/注释/配置格式
- 仅新增类型定义（未实现）
- 上一个 task 刚触发过 QA 且当前变更极小

## 决策输出（强制，三选一）

**触发：**

```text
🧪 触发 QA — 原因: {理由}
   累积变更: {N} 个 task | 风险评估: {总分}
   调用 tc-qa-engineer...
   {QA 完成:}
   ✓ QA 通过 / ⚠ 发现 {N} 个问题，已修复 → 复审通过
```

**跳过（评分未达阈值）：**

```text
⏭ 跳过 QA — 评分 {总分}/20 未达阈值
   原因: {简述, 如"单文件 UI 改动，风险低"}
```

**跳过（skill 未注册）：**

```text
⏭ 跳过 QA — tc-qa-engineer 未注册
   建议手动测试本 task
```

## 退出消息

```text
→ 进入 N7 上下文管理
```

QA 通过 → 继续。发现问题 → 修复后重新 QA **1 轮**。1 轮复测仍失败 → **暂停并报告**，交用户裁决，不再自动循环。

> 这里限制为 1 轮是有意为之：QA 本身已包含"读 specs + 设计用例 + 跑测 + 修复"完整 pipeline，每多一轮接近线性放大 token 消耗。1 轮复测足以兜住"自己刚改坏的小 bug"，更复杂的失败应让人介入而非让 AI 反复试。
