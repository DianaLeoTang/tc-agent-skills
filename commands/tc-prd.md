# /tc-prd — 需求文档 → 开发规格生成

支持两种模式：新建需求和需求变更。

> 💡 **本命令的输入是 `docs/feature-N-{name}.md`**。如果你的需求还在「混沌阶段」（一句话、思路未清晰、不知道是 1 个还是 N 个 feature），**先跑 [`/tc-discuss`](tc-discuss.md)** 把需求聊清晰，让它生成 docs/feature 文档后再回来跑本命令。
>
> 完整工作流：`/tc-init`（一次性）→ `/tc-discuss`（聊需求）→ `/tc-prd`（本命令）→ `/tc-ai`（实施）→ `/tc-test`（跑测试）

## 输入参数

`$ARGUMENTS` 格式：

- **新建模式**：`/tc-prd {项目文件夹路径}`
- **变更模式**：`/tc-prd --change {N}.{feature-name} 变更内容描述`

用户提供一个项目文件夹路径，文件夹结构约定：

```text
{项目文件夹}/
├── docs/           ← 需求文档（必须存在，PRD 从这里读取）
├── 1.xxx/          ← 已有的 specs（如有）
├── 2.xxx/          ← 本次生成的 specs
└── ...
```

## 模式判断

如果 `$ARGUMENTS` 以 `--change` 开头 → 进入变更模式（跳到「变更模式」章节）
否则 → 进入新建模式

---

## 新建模式

### Step 1: 解析输入，读取需求文档

从 `$ARGUMENTS` 提取项目文件夹路径，记为 `SPECS_DIR`。

读取 `{SPECS_DIR}/docs/` 下的所有文件作为需求源：

- 支持 `.md`、`.txt`、`.pdf` 等文档格式
- 如果 docs/ 下有多个文件，全部读取并综合分析
- 如果 docs/ 不存在或为空，报错提示用户先在 docs/ 下放入需求文档

### Step 2: 获取项目名称

- 从当前目录的 `package.json` name 字段、`Cargo.toml`、`go.mod` 等提取项目名
- 如无法提取，使用当前目录名
- 转为 kebab-case，记为 `PROJECT_NAME`

### Step 3: 探测项目架构类型

扫描项目根目录、配置文件、目录结构、依赖声明，自行判断架构类型（monorepo / 多仓库 / 单体应用 / Web3 等）。记录 `ARCH_TYPE`。

### Step 4: 读取项目上下文

- 读取各仓库的 `.claude/CLAUDE.md` 了解技术栈
- 读取 `.claude/rules/` 下所有规则文件
- 扫描目录结构，了解现有模块划分

### Step 5: 分析需求

从文档中提取功能目标、用户故事、验收标准、约束条件、依赖。

### Step 5.5: 开放问题确认（含技术决策）

分析需求后，如果存在以下情况，**必须暂停并与用户对话确认**，不要自行假设。**技术 / 重构 / 基础设施类 feature 也要走这一步**——不允许"技术决策由我自行选定"的逃避：

**业务方面：**

- 需求描述模糊或有歧义的功能点
- 缺少关键信息（如目标平台、兼容性要求、第三方服务选型）
- 业务逻辑有矛盾或不完整
- 涉及权限、支付、敏感操作等需要明确确认的功能

**技术决策方面（重构 / 基础设施 / 测试体系类 feature 必查）：**

- 多种技术实现方案差异较大（如 TS 严格度 A/B/C、HTTP 客户端 axios vs fetch）
- 工具选型决策（Vitest vs Jest、Playwright vs Cypress、zod vs valibot）
- 量化阈值（覆盖率门槛百分比、超时时间、并发数等数值）
- 引入新依赖（任何 npm/pip 新包都应让用户知晓）
- API 契约变更（response_model / 字段增删 / status code 变更）

格式：

```
❓ 需要确认以下问题：

1. [Q-001] {问题描述} — {为什么需要确认} — 默认建议：{方案 A/B}
2. [Q-002] {问题描述} — {为什么需要确认} — 默认建议：{方案 A/B}

请逐一回复后继续生成 specs（回复"全默认"即采用所有建议）
```

所有问题确认完毕后再进入 Step 6。**已决议的 Q-XXX 写入 requirements.md 的「开放问题」章节并标 `✓ 决议`，保留决策痕迹**。

### Step 6: 推断 feature 名称

根据需求内容生成一个简洁的 kebab-case 英文名称。

### Step 7: 生成 specs 目录（三件套不可缺）

检查 `{SPECS_DIR}/` 下已有的编号目录（如 `1.xxx/`、`2.xxx/`），取最大编号 +1。

```text
{SPECS_DIR}/
├── docs/                        ← 需求文档（输入）
├── 1.比如这是一个已有的标题/     ← 已有 specs
└── 2.{feature-name}/            ← 本次新建
    ├── requirements.md          ⚠️ 必须有
    ├── design.md                ⚠️ 必须有
    └── tasks.md                 ⚠️ 必须有
```

⚠️ **强制规定（不可绕过）**：

- **每个 feature 必须生成完整三件套**，无例外
- **包括纯技术 / 重构 / 基础设施 / 测试体系类 feature**（如 TS 迁移、加 response_model、搭测试基础设施）
- 不允许「简化版 specs（只有 design + tasks，无 requirements）」「纯技术 feature 没业务需求」等借口跳过 requirements.md
- 技术 feature 的 requirements.md 把 F-XXX 写成技术性需求即可（详见 Step 8 模板示例）
- 历史教训：项目初期 0.5/0.6/0.7 三个技术 feature 跳过 requirements.md，导致演进背景与验收标准散落各处，事后必须 backfill

### Step 8: 生成 requirements.md

```markdown
# {Feature 名称} — 需求规格

## 概述

{一句话描述：要做什么 + 解决什么问题}

## 项目信息

- 项目名: {PROJECT_NAME}
- 架构类型: {ARCH_TYPE}
- 改造目录: {如仅前端则 frontend/，仅后端则 backend/}
- ROADMAP 引用: {如有}
- 上游 / 下游 feature: {如有依赖关系}

## 需求版本

| 日期         | 版本 | 说明     |
| ------------ | ---- | -------- |
| {YYYY-MM-DD} | v1   | 初始需求 |

## 演进背景（WHY）

{为什么要做这个 feature。}

业务 feature：写业务价值 / 用户痛点 / 商业目标。
技术 feature：写技术债动因 / 维护成本拐点 / 工程质量动因。

如有用户原话或 memory feedback，引用过来：
> 用户原话："xxx"（参考 [memory](~/.claude/projects/.../memory/feedback_xxx.md)）

## 用户故事 / 受益方

业务 feature 示例：
- 作为**付费用户**，我想 {功能}，以便 {价值}
- 作为**管理员**，我想 {功能}，以便 {价值}

技术 feature 示例：
- 作为**前端开发**，我想 {能力}，避免 {痛点}
- 作为**Reviewer**，我想 {能力}，便于 {决策}
- 作为**新加入工程师**，我想 {能力}，加速 {上手}

## 功能需求

业务 feature 示例：
1. [F-001] 用户可用邮箱+密码登录
2. [F-002] 登录失败显示中文错误提示

技术 feature 示例（同样合法）：
1. [F-001] tsconfig 启用 strict: true，不允许 noImplicitAny: false 兜底
2. [F-002] 所有 axios 响应必须使用 paths[...] 派生类型，不允许手写 interface
3. [F-003] 测试覆盖率门槛 lines ≥ 70% / branches ≥ 60%
4. [F-004] schema snapshot 测试存在，防 api.ts 静默漂移

## 非功能需求

- 性能: {要求，如 P95 ≤ 500ms / 测试套件 < 5min}
- 安全: {要求}
- 兼容性: {要求，如 SQLite + PG 同时跑通}
- 可维护性: {要求}

## 验收标准

每个 AC 必须**可机器验证**（命令退出码、覆盖率数字、文件存在性等）：

- [ ] [AC-001] {可验证标准，如「pytest -v 全绿」「tsc --noEmit EXIT=0」「覆盖率 lines ≥ 70%」}
- [ ] [AC-002] {...}

## 依赖

- 既有 / 上游：{已有依赖}
- 新引入：{本 feature 新增的依赖}

## 不做（明确划界）

显式列出**本 feature 不涉及**的范围，避免后续讨论"这归本 feature 吗"无据可查：

- ❌ 不做 {范围 A}
- ❌ 不做 {范围 B}（属 future feature N）
- ❌ 不改 {模块 X}

## 开放问题

- [Q-001] {问题描述} — {决议状态：待确认 / ✓ 决议: 选择 X 因为 Y}
- [Q-002] {...}
```

### Step 9: 生成 design.md

**必须先读取项目 `.claude/CLAUDE.md` 和 `.claude/rules/` 下所有规范文件**，设计方案必须遵循项目已有的技术规范和约定。

按功能模块设计，每个模块说明涉及哪些层（前端、后端、数据库、合约等），具体分层根据项目实际架构决定，不做硬编码限制。

```markdown
# {Feature 名称} — 技术设计

## 设计版本

| 日期         | 版本 | 说明     |
| ------------ | ---- | -------- |
| {YYYY-MM-DD} | v1   | 初始设计 |

## 项目架构

- 架构类型: {ARCH_TYPE}
- 涉及层: {根据项目实际情况列出}

## 功能模块设计

### 模块 1: {模块名}

{技术方案，遵循 .claude/rules/ 中的规范}

**涉及层及关键设计:**

{根据项目实际分层描述，如数据模型、API 接口、组件设计、合约接口等}

### 模块 2: {模块名}

...

## 接口契约

{API、RPC、合约接口等 — 根据项目类型决定}

## 数据模型

{数据表/模型/链上存储 — 根据项目类型决定}

## 安全考虑

{基于 .claude/rules/security.md 和项目特有的安全规范}

## 技术决策

| 决策 | 选项 | 理由 |
| ---- | ---- | ---- |
```

### Step 10: 生成 tasks.md

**按功能拆任务。** AI 执行时根据 design.md 自动判断每个任务涉及哪些层。

```markdown
# {Feature 名称} — 任务清单

## 任务版本

| 日期         | 版本 | 说明     |
| ------------ | ---- | -------- |
| {YYYY-MM-DD} | v1   | 初始任务 |

## 项目信息

- 项目名: {PROJECT_NAME}
- 架构类型: {ARCH_TYPE}
- specs 路径: {SPECS_DIR}/{N}.{feature-name}/

## 任务列表

### 功能 1: {功能名}

- [ ] T-001: {任务描述} ~{预估时间}
- [ ] T-002: {任务描述} ~{预估时间}

### 功能 2: {功能名}

- [ ] T-003: {任务描述} ~{预估时间}

### 集成与测试

- [ ] T-010: 联调测试 ~{预估时间}
- [ ] T-011: E2E 测试 ~{预估时间}

## 依赖关系

- T-002 依赖 T-001

## 风险点

- {可能遇到的问题及应对}
```

**任务拆解原则：**

- 按功能拆，AI 执行时读 design.md 自动识别涉及哪些层
- 原子性，可独立完成和验证
- 预估完成时间（5min / 15min / 30min / 1h）
- **粒度控制**：每个子 specs（feature 目录）不宜过大，单个 tasks.md 控制在 **10-15 个任务以内**。如果需求过大，应在 Step 6 之前拆成多个独立的 feature 目录（如 `2.user-auth-login`、`3.user-auth-register`），每个 feature 有自己的 requirements/design/tasks 三件套。这样 tc-ai 执行时上下文可控，不会因为 specs 太大导致丢失关键信息。

### Step 11: 输出总结 + 自检 checklist

**先**完成自检（任何一项不达标都要回头补）：

- [ ] requirements.md 存在且包含全部六大块：**演进背景 WHY** / **用户故事** / **F-XXX 功能需求** / **AC-XXX 验收标准** / **不做（NOT IN SCOPE）** / **开放问题**
- [ ] design.md 包含「项目架构」「功能模块设计」「接口契约」「数据模型」「技术决策」「不做」「风险」
- [ ] tasks.md 任务数 ≤ 15（超过应在 Step 6 之前拆 feature）；每个任务有时间估算
- [ ] AC-XXX 都是**可机器验证**的（命令 EXIT=0、覆盖率数字达标、文件存在等）
- [ ] 三件套互相引用：design 引用 requirements 的 F-XXX；tasks 引用 design 的模块名
- [ ] Step 5.5 的开放问题已得到用户回复并写入 requirements.md「开放问题」章节（带 ✓ 决议）

**后**生成总结报告：

- Feature 名称和序号
- Specs 路径
- 涉及的技术层
- 总任务数和预估总时间
- 自检 checklist 全过 ✅
- 提示用户：**请审查 specs 后，运行 `/tc-ai` 开始开发**

## 变更模式

当输入 `/tc-prd --change {N}.{feature-name} 变更内容` 时执行。

### Step C1: 定位已有 specs

在 `{SPECS_DIR}/` 中查找匹配的编号目录。

如果只传了序号（如 `--change 1`），自动匹配 `1.*` 目录。

找不到则报错提示。

### Step C2: 读取现有 specs

读取该目录下的：

- `requirements.md` — 当前需求
- `design.md` — 当前设计
- `tasks.md` — 当前任务（注意哪些已完成 `[x]`）

### Step C3: 解析变更内容

变更内容可以是：

- 纯文本描述变更
- 文件路径（新的需求文档）
- URL

### Step C4: 对比分析

将变更内容与现有 requirements.md 对比，识别：

- **新增**的功能需求
- **修改**的功能需求
- **删除**的功能需求
- 对验收标准的影响

### Step C5: 更新 requirements.md（含 AC-XXX / Q-XXX 演进）

- 在「需求版本」表中追加新版本行：

```markdown
## 需求版本

| 日期       | 版本 | 说明             |
| ---------- | ---- | ---------------- |
| 2026-04-11 | v1   | 初始需求         |
| 2026-04-15 | v2   | 新增微信登录方式 |
```

- 在「功能需求」中标注变更：

```markdown
## 功能需求

1. [F-001] 邮箱 + 密码登录
2. [F-002] 手机号 + 验证码登录
3. [F-003] Tab 切换登录方式
4. [F-004] ~~短信验证码 5 分钟过期~~ `[v2 删除]`
5. [F-005] 微信扫码登录 `[v2 新增]`
6. [F-006] 表单验证 `[v2 修改: 新增微信回调校验]`
```

- 在「验收标准」中同步标注变更（不只 F 要追踪，AC 也要）：

```markdown
## 验收标准

- [x] [AC-001] 邮箱+密码登录成功后跳转主页
- [ ] [AC-002] 手机号验证码 60 秒倒计时 `[v2 调整: 去掉倒计时，改即时校验]`
- [ ] [AC-005] 微信回调成功后展示昵称 `[v2 新增]`
```

- 在「开放问题」中保留**所有历史 Q-XXX 含决议状态**（不删除已决议项，保留决策痕迹）：

```markdown
## 开放问题

- [Q-001 ✓ 决议 v1] 短信通道选哪家？— 阿里云
- [Q-002 ✓ 决议 v2] 微信登录方式：扫码 / 公众号？— 扫码（OAuth2.0）
- [Q-003 v2 新增] 微信账号绑定多手机号场景如何处理？— 待确认
```

### Step C6: 更新 design.md

- 追加设计版本
- 新增/修改受影响的功能模块设计
- 不动未受影响的模块
- 标注变更原因

### Step C7: 更新 tasks.md

- 追加任务版本
- 已完成的任务 `[x]` 保持不动
- 受变更影响的未完成任务标记 `[CHANGED]` 并更新描述
- 因变更作废的任务标记 `[DROPPED]`
- 新增的任务标记 `[NEW]`

```markdown
## 任务列表

### 功能 1: 登录表单

- [x] T-001: 创建登录页面和 Tab 切换组件 ~30min
- [x] T-002: 实现邮箱登录表单 ~30min
- [ ] T-003: 实现手机号登录表单 ~30min `[CHANGED v2: 去掉倒计时]`
- [ ] ~~T-004: 短信过期逻辑~~ `[DROPPED v2]`

### 功能 2: 微信登录 `[NEW v2]`

- [ ] T-008: [NEW] 微信 OAuth 回调处理 ~30min
- [ ] T-009: [NEW] 微信登录按钮组件 ~15min
```

### Step C8: 输出变更摘要

```
📝 需求变更: 1.user-auth v1 → v2

新增: 2 个功能需求, 2 个任务
修改: 1 个功能需求, 1 个任务
删除: 1 个功能需求, 1 个任务
未受影响: 3 个已完成任务保持不动

请产品审查变更后，运行 /tc-ai 继续开发（会跳过已完成任务）
```

## 示例用法

```bash
# 新建需求 — 提供项目文件夹路径，docs/ 下放需求文档
/tc-prd ~/projects/my-app-specs
/tc-prd /path/to/project-specs

# 需求变更 — 同样基于项目文件夹，指定要变更的 feature
/tc-prd --change 1.user-auth 新增微信扫码登录方式
/tc-prd --change 2 ~/projects/my-app-specs
```
